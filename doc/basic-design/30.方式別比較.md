Salesforceからデータを抽出しS3へ保存する処理において、**Amazon AppFlow方式**と**AWS Lambda方式（標準ライブラリのみ）**のメリット・デメリットを整理しました。

---

# Salesforce抽出バッチ：方式別比較

## 1. Amazon AppFlow 方式

GUIベースで設定するノーコード/ローコードの連携方式です。

| 項目 | メリット | デメリット |
| --- | --- | --- |
| **開発コスト** | **極めて低い**。ブラウザ上の数クリックで設定が完了し、コードを書く必要がない。 | 複雑なデータ変換や、動的な条件分岐などの高度なロジックは実装しにくい。 |
| **認証管理** | **フルマネージド**。アクセストークンの更新やリフレッシュ処理をAWSが自動で行う。 | 初回の接続設定時に、開発者による手動のログイン認可操作が必要。 |
| **保守運用** | **低い**。ライブラリの脆弱性対応やバージョンアップを意識する必要がない。 | 実行ごとにAppFlowの利用料金（転送量・実行回数）が発生する。 |
| **標準機能** | フィルタリング、マスキング、**増分転送（差分抽出）**が標準機能として提供。 | AWSが対応しているオブジェクトや機能の範囲内に限定される。 |

---

## 2. AWS Lambda 方式（標準ライブラリのみ）

Python等の標準ライブラリ（`urllib`等）を用いてAPIを直接叩く方式です。

| 項目 | メリット | デメリット |
| --- | --- | --- |
| **柔軟性** | **無限大**。取得データの複雑な加工、複数APIの組み合わせ、外部システムへの同時通知などが自由自在。 | 認証ロジック（JWT署名）からAPI呼び出しまで、すべて自前で実装・テストが必要。 |
| **セキュリティ** | **最高レベル**。オープンソース（OS）に依存しないため、OSライブラリ由来の脆弱性リスクがゼロ。 | 秘密鍵の管理やJWT生成ロジックの堅牢性を自前で担保しなければならない。 |
| **コスト** | **安価**。Lambdaの実行時間とリクエスト数に応じた課金のみで、AppFlowの基本料がかからない。 | 開発工数（人件費）と、APIの仕様変更に伴うコードメンテナンスコストがかかる。 |
| **制御** | Bulk API 2.0 のジョブ監視やエラーハンドリングを細かく制御できる。 | Lambdaの実行時間制限（最大15分）を考慮した非同期設計が必要。 |

# どちらを採用すべきか？（判定基準）

### **Amazon AppFlow を選ぶべきケース**

* **スピード優先:** とにかく早く、安定したデータ連携を構築したい。
* **標準的な抽出:** 特定のオブジェクトをそのまま、あるいは簡単なフィルタでS3に持っていきたい。
* **保守リソース不足:** 専任の開発者がおらず、インフラ側の設定で運用を回したい。
* **増分抽出:** Salesforce側の「最終更新日時」に基づいた差分抽出を楽に実装したい。

### **AWS Lambda（標準ライブラリ）を選ぶべきケース**

* **高度な加工:** S3に保存する前に、データの計算、複数テーブルの結合、匿名化などを複雑に行いたい。
* **セキュリティ制約:** 外部ライブラリの使用が厳格に禁止されており、自社コードのみで完結させる必要がある。
* **複雑なワークフロー:** 抽出後に別のシステムをキックするなど、前後の処理と密接に連携させたい。
* **大量のオブジェクト:** 多数のオブジェクトを動的にループ処理で抽出したい（AppFlowはフローごとの設定が必要）。

## 次のステップ

どちらの方式にするか方針は固まりましたか？

* **AppFlowの場合**: 「増分転送（差分抽出）」を有効にするための具体的な設定箇所を整理できます。
* **Lambdaの場合**: 標準ライブラリだけで認証を通すための「JWT生成コードの骨組み」を作成できます。
