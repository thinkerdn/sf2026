# 詳細設計書：S3データ取り込み・Salesforce反映バッチ (AppFlow)

## 1. ワークフロー概要

S3バケットの特定パスに配置されたCSV/JSONファイルを検知し、AppFlowを用いてSalesforceのレコードを更新・作成（Upsert）する。

## 2. 接続・認証設計 (Connection)

* **接続名**: `S3_TO_SF_CONNECTION`
* **認証方式**: OAuth 2.0 (認可コードフロー)
* 抽出バッチと共通の「接続アプリケーション (Connected App)」を利用可能。
* **権限**: 書き込み対象のオブジェクトに対する「作成」「更新」「削除（必要に応じて）」の権限を持つAPI専用ユーザーで認可。



## 3. フロー詳細設計 (Flow Configuration)

### 3.1 送信元設定 (Source: S3)

* **ソースバケット**: `[Your-Inbound-S3-Bucket-Name]`
* **ソースパス**: `inbound/salesforce/[ObjectName]/`
* **ファイル形式**: CSV (カンマ区切り、ヘッダーあり)
* **転送モード設定**:
* **増分転送 (Incremental transfer)**: S3上の新しいファイル（タイムスタンプが前回実行以降のもの）のみを対象とする。
* **全件転送 (Full transfer)**: マスタデータの初期投入や再同期の際に、フォルダ内の全ファイルを処理するために切り替え可能。



### 3.2 送信先設定 (Destination: Salesforce)

* **オブジェクト**: ターゲットとなるSalesforceオブジェクト（例: `Account`, `Custom_Object__c`）
* **書き込みメソッド**: **Upsert**
* **マッチングキー**: Salesforce側で設定した **外部ID (External ID)** フィールドを指定。
* これにより、S3側のデータにSalesforceのIDが含まれていなくても、一意のキー（顧客コード等）で既存レコードの更新と新規登録を自動判別。


* **API種別**: **Bulk API 2.0**
* 大量データ更新時のガバナ制限（API制限）を最小限に抑え、処理効率を最大化。



### 3.3 データマッピング (Mapping)

* **マッピングルール**: S3のCSVカラム名とSalesforceのAPI参照名を紐付け。
* **データバリデーション**:
* 日付形式、数値型などの型チェック。
* マッピングされない項目がある場合、または無効な値がある場合の挙動（レコードをスキップ）を設定。



## 4. エラーハンドリングとリカバー設計

* **エラーレポート設定**:
* Salesforce側の入力規則（バリデーションルール）や重複エラーで失敗したレコードを特定するため、**「エラー出力用のS3バケット」**を指定。
* パス: `error-log/salesforce/[ObjectName]/YYYY/MM/DD/`
* 出力内容: 失敗したレコードの内容 ＋ エラー理由（ErrorCode, ErrorMessage）。


* **監視通知**:
* EventBridgeで「Flow Execution Status = Failure」を監視し、SNS経由で管理者に即時通知。



## 5. 運用設計

* **ファイルのアーカイブ**:
* 取り込み完了後、AppFlowの機能（またはS3ライフサイクル/Lambda）により、ソースファイルを「処理済み（archived）」フォルダへ移動させる運用を推奨。


* **再実行手順**:
* エラーレポートを確認し、ソースファイルを修正後、再度S3にアップロードすることで自動的に次回のスケジュールで再取り込みが行われる。



## 6. セキュリティ設計

* **IAMロール**: AppFlowに対し、ソースバケットの `GetObject` およびエラー出力バケットへの `PutObject` 権限を付与。
* **暗号化**: S3上のファイルおよび通信経路（TLS 1.2+）の暗号化。
