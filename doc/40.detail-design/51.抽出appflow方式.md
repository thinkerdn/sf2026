# 詳細設計書：Salesforceデータ抽出・S3保存バッチ (AppFlow)

## 1. ワークフロー概要

AppFlowの標準コネクタを利用し、Salesforceのオブジェクトデータを抽出後、S3バケットへパーティショニングされた状態で保存する。

## 2. 接続・認証設計 (Connection)

* **接続名**: `SF_TO_S3_CONNECTION`
* **認証方式**: OAuth 2.0 (認可コードフロー)
* Salesforce側で「接続アプリケーション (Connected App)」を作成し、`Client ID` と `Client Secret` を発行。
* AppFlowの接続設定画面で認可を行い、リフレッシュトークンをAWS側に保持させる。


* **コネクタプロファイル**: AWS Secrets Manager に認証情報が自動的に格納・管理される。

## 3. フロー詳細設計 (Flow Configuration)

### 3.1 送信元設定 (Source)

* **送信元システム**: Salesforce
* **オブジェクト**: 任意のオブジェクト (例: `Account`, `Opportunity` 等)
* **API種別**: **Bulk API 2.0** を優先選択
* APIコール数の節約と、Lambdaタイムアウトを考慮する必要がないスケーラビリティを確保。



### 3.2 送信先設定 (Destination)

* **送信先システム**: Amazon S3
* **バケット名**: `[Your-S3-Bucket-Name]`
* **バケットプレフィックス**: `extract/[ObjectName]/`
* **ファイル形式設定**:
* 形式: CSV (または分析用途に応じて Parquet)
* 集約: 「ファイルを結合する」を選択（大量の小さなファイル生成を防止）


* **パーティショニング設定**:
* タイムスタンプ（実行日）に基づくフォルダ分割を有効化。
* 例: `.../year=2023/month=10/day=27/`



### 3.3 実行タイミングと転送モード (Trigger & Transfer Mode)

* **トリガー**: スケジュール実行
* **転送モード設定**:
1. **通常運用 (増分転送)**:
* `Incremental transfer` を選択。
* 判定フィールド: `SystemModStamp` または `LastModifiedDate`。
* 初回実行時は全件、2回目以降は前回実行時からの差分のみを取得。


2. **メンテナンス用 (全件転送)**:
* 全件再投入が必要な場合は、一時的に `Full transfer` へ変更して手動実行またはスケジュール実行。





## 4. データ加工・マッピング設計 (Mapping)

* **マッピング方式**: 「すべてのフィールドを直接マッピングする」または個別に指定。
* **検証タスク**:
* データ不備（必須項目漏れなど）がある場合、レコードをスキップするかフローを停止するかを選択。


* **フィルタ設定**: 必要に応じて、削除フラグ (`isDeleted = false`) などの条件を追加。

## 5. 監視・エラーハンドリング設計

* **CloudWatch Logs**: フローの実行詳細（成功/失敗件数）を記録。
* **EventBridge 通知**:
* イベントパターン: `AppFlow Flow Execution Status Change`
* アクション: SNS経由で管理者にメール/Slack通知。


* **エラーレコードの出力**: 取り込みエラーが発生した場合、指定したS3バケットにエラーログを出力するよう設定。

## 6. セキュリティ設計

* **データの暗号化**: S3バケット側で SSE-KMS (AWSマネージド型鍵) を強制。
* **IAMポリシー**:
* AppFlow サービスロールに対し、対象のS3バケットへの `PutObject`, `GetBucketLocation` 権限のみを付与。



---

## 7. 運用上の考慮事項

* **Salesforce側の設定**: API専用の「統合ユーザー (Integration User)」を使用し、パスワードの有効期限を無期限に設定するか、適切なリフレッシュ管理を行うこと。
* **増分の重複回避**: S3側で後続処理（Athena等）を行う場合、重複が発生しないようファイル名やタイムスタンプによるユニーク管理を確認すること。
