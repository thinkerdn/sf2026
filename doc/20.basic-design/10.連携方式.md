# AWS (Lambda/AppFlow) と Salesforce の連携方式比較

## 1. AWS Lambda による連携（カスタムコード方式）

プログラムを記述して、SalesforceのAPI（REST/Bulk/Soap）を直接呼び出す方式です。

| 項目 | 内容 |
| --- | --- |
| **主な認証方式** | OAuth 2.0 JWT ベアラフロー（推奨）、クライアント認証フロー |
| **連携タイミング** | EventBridgeによるスケジュール実行、または特定イベントによるトリガー実行 |
| **メリット** | ・**極めて高い柔軟性**: 複雑なビジネスロジックや高度なデータ変換が可能。<br>

<br>・**オープンソース非依存**: 標準ライブラリのみで構成すれば、外部依存による脆弱性リスクを排除できる。<br>

<br>・**きめ細やかな制御**: APIの呼び出し順序やリトライロジックを完全に制御可能。 |
| **デメリット** | ・**開発・保守コスト**: コードの記述、テスト、セキュリティ対策（JWT署名等）の実装が必要。<br>

<br>・**ガバナ制限の意識**: API制限やLambdaのタイムアウト（15分）を考慮した設計が必要。 |

---

## 2. Amazon AppFlow による連携（マネージド方式）

AWSが提供するノーコード/ローコードのデータ転送サービスを利用する方式です。

| 項目 | 内容 |
| --- | --- |
| **主な認証方式** | OAuth 2.0（認可コードフロー / AppFlowによるフルマネージド管理） |
| **連携タイミング** | スケジュール実行、オンデマンド実行、またはイベント（S3配置など）実行 |
| **メリット** | ・**ノーコードで高速導入**: GUI設定のみで接続が完了し、開発工数を大幅に削減できる。<br>

<br>・**運用負荷の軽減**: 認証トークンの更新やスケーリング、API制限への対応をAWSが代行。<br>

<br>・**標準機能の充実**: 増分転送（差分抽出）やフィルタリング、マッピング機能が標準搭載。 |
| **デメリット** | ・**カスタマイズの制限**: 複雑な計算や、複数オブジェクトを跨ぐ高度なロジックは不向き。<br>

<br>・**追加コスト**: 実行ごとにAppFlowの利用料金が発生する（データ量が多い場合に注意）。 |

---

## 3. どちらを選択すべきか？（選定基準）

| シナリオ | 推奨される連携方式 |
| --- | --- |
| **単純なデータの同期・バックアップ** | **Amazon AppFlow**<br>

<br>（開発コストを抑え、AWSのマネージド機能に任せる） |
| **増分・全件の切り替えを頻繁に行う** | **Amazon AppFlow**<br>

<br>（標準機能で切り替えが容易） |
| **取得データの加工や複雑な計算が必要** | **AWS Lambda**<br>

<br>（プログラムによる自由な処理が必要） |
| **セキュリティ制約で外部ライブラリ禁止** | **AWS Lambda**<br>

<br>（標準ライブラリのみで自作可能） |

---

## 4. 共通のベストプラクティス

1. **セキュリティ**: 認証情報（クライアントID、秘密鍵など）は必ず **AWS Secrets Manager** で管理し、環境変数への直書きを避ける。
2. **監視**: **CloudWatch Alarms** を設定し、Lambdaの失敗やAppFlowのフロー停止を即座に検知・通知する。
3. **API制限の考慮**: SalesforceのAPIコール数は有限であるため、可能な限り **Bulk API 2.0** を活用して効率的に通信を行う。
